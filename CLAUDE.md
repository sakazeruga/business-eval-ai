# 事業評価AIサービス 設計仕様書
> Claude Code実装用 / 最終更新: 2026-06-16

---

## 1. プロジェクト概要

### サービスコンセプト
事業構想を入力すると、構造化フレームワークで多角的分析を行い、現状評価と改善案を出力するAIサービス。

### ターゲット・展開先（クローズド）
| モード | 対象 |
|---|---|
| **スタートアップ評価モード** | ジーズアカデミー / Globisアントレプレナークラブ / VC評価支援 |
| **COOターンアラウンドモード** | 自社COO代行 / COO代行養成講座レンタル用 |

---

## 2. モード切り替え設計

```
[入力画面] → モード選択 → [スタートアップ評価] or [COOターンアラウンド]
                                  ↓                        ↓
                          Why me/Why now評価          財務遡及分析
                          市場成長性分析              オペレーション診断
                          NPV/EXIT算出               穴塞ぎ優先設計
```

### モード別の差分

| 評価項目 | スタートアップ | COOターンアラウンド |
|---|---|---|
| Why me / Why now | ✅ 必須 | ❌ 不要（既存事業に聞かない） |
| 市場成長性 | ✅ | ✅（ただし遡及的視点） |
| VRIO分析 | ✅ | ✅ |
| CF分析 | ✅（将来予測） | ✅（原因遡及が主軸） |
| NPV / EXIT価値 | ✅ 起業家向け | 🔄 再建後企業価値として転用 |
| クロスSWOT | ✅ | ✅ |
| 優先度マトリクス | Eisenhower（将来） | Eisenhower（穴塞ぎ→アクセル→フロー→文化） |
| Why me重み | 1次ドア重視 | 2次ドア（実行ネットワーク）重視 |

## 3. 評価フレームワーク（スタートアップモード）

### 3-1. 現状分析

#### ① Why me / Why now パワー
→ 詳細は「5. Why me / Why now 設計」参照

#### ② 市場成長性
**Why Now = 現状成長性 × 潜在成長性（爆発力） × 社会還元性（調達力）**

| 因子 | 定義 | 評価方法 |
|---|---|---|
| 現状成長性 | TAM/SAMの直近成長率 | 市場データ（Gemini web検索） |
| 潜在成長性（爆発力） | S字カーブの屈折点・ネットワーク効果・プラットフォーム性 | 定性評価 |
| 社会還元性（調達力） | 社会的インパクト → 補助金・ESG・グラント調達可能性 | 政策・トレンド調査 |

> **Deep Tech / ソーシャル事業**は潜在成長性・社会還元性を別途厚く評価すること

#### ③ 市場適合性（VRIO構築）
- Value / Rarity / Imitability / Organization の4軸評価
- 競合他社との比較マッピング

#### ④ マーケティング施策評価
- 顧客獲得チャネルの蓋然性
- CAC / LTV 試算
- GTM戦略の実現性

#### ⑤ オペレーション蓋然性
- チーム構成・実行体制
- クリティカルパスの特定
- ボトルネックリスク

#### ⑥ 財務戦略評価（キャッシュフロー循環）
- 収益モデルの構造
- CF循環の健全性
- 資金調達計画との整合性

### 3-2. 総合評価
- **NPV算出**（割引率・成長率・期間を入力から推定）
- **EXIT事業価値**（PER/PSR/DCFベース、業種別マルチプル適用）

### 3-3. 改善案

| 出力項目 | 内容 |
|---|---|
| クロスSWOT分析 | SO/ST/WO/WT戦略の導出 |
| 追加ソリューション提案 | ヒト・モノ・カネ・情報のリソース調達案 |
| 身に着けるべき習慣 | 創業者個人へのアクションアイテム |
| クリティカルシンキングによる現状打開策 | 論理的現状打破の優先3手 |

## 4. 評価フレームワーク（COOターンアラウンドモード）

### 分析の基本思想
**財務から根本原因を逆算 → ビジネスモデル分解 → Eisenhower優先度で打ち手を決める**

### 優先度マトリクス（固定順）
```
① 穴を塞ぐ（キャッシュアウト防止・緊急課題）
② アクセラレーターを加える（成長ドライバーの特定）
③ フロー速度を改善する（オペレーション効率化）
④ 文化を作る（持続的組織能力の構築）
```

### 出力構成
- 財務診断（何が壊れたか・いつから・なぜか）
- オペレーション問題マップ
- 優先打ち手リスト（即時/30日/90日）
- VRIO・クロスSWOT（共通モジュール流用）
- リソース調達案（ヒト・モノ・カネ・情報）

## 5. Why me / Why now 詳細設計

### Why Now評価（再掲・市場成長性と統合）
```
Why Now スコア = 現状成長性 × 潜在成長性 × 社会還元性
```
※ 乗算モデル = 一因子ゼロなら全体ゼロ（意図的設計）

### Why Me評価
**3因子の乗算スコア**

```
Why Me スコア = 情報収集能力 × 背景（継続性・逆境耐性） × 巻き込み力
```

## 6. 入力フォーム設計（質問設計）

### Why Me 評価用質問（3問）

#### Q1. 情報収集能力
```
「この事業領域の主要競合を5社と、それぞれの差別化ポイントを
 今すぐ挙げてください。」
```
→ 即答できるか・精度はどうかで情報収集能力を評価

#### Q2. 背景（継続性・逆境耐性）
```
「誰の助けも借りずに、最も長く取り組み続けたことは何ですか？
 期間と、何度やめようと思いましたか？」
```
> **評価注意点**: 「挫折回数0」は2パターンある
> - 偽陽性: 大きなリスクを取っていないため失敗していない
> - 本物: 失敗を過程として再定義できる認知能力（→実績と突合して判定）

#### Q3. 巻き込み力（★ 核心質問）
```
「今すぐ理由を聞かずに1万円を貸してくれそうな、
 ハブとなる人は何人いますか？
 また、その人たちのネットワーク重複はどのくらいですか？」
```
追加質問:
```
「その中で、あなた自身では開けられない扉を直接開けてくれる人（1次ドア）と、
 複数のコミュニティを橋渡しできる人（2次ドア）はそれぞれ何人ですか？」
```

### 巻き込み力 スコアリングルーブリック

| スコア | 定義 |
|---|---|
| 1 | 1万円貸してくれる人がいない |
| 2 | 重複ありの信頼ネットワークのみ |
| 3 | 非重複の2次ドア保有者が複数いる |
| 4 | 1次ドアか2次ドアどちらかが強い |
| **5** | **1次ドア＋2次ドアを両方持ち、重複が少ない** |

**1次ドア / 2次ドアの定義**
| 種別 | 定義 | 例 |
|---|---|---|
| 1次ドア | 資源への直接アクセス（権限・資金・信用） | 上場企業元役員、VC、行政決裁者 |
| 2次ドア | ネットワーク拡張速度（多様コミュニティのブローカー） | フットワーク軽い同世代、異業種ハブ人材 |

**重み設定（モード別）**

| 因子 | スタートアップ | COOターンアラウンド |
|---|---|---|
| 1次ドア | 高（EXIT・調達に直結） | 中 |
| 2次ドア | 中 | 高（実行ネットワーク重視） |

## 7. 技術アーキテクチャ

### モデル構成（マルチモデルアンサンブル）

```
[入力: 事業構想テキスト + フォーム回答]
        ↓
┌─────────────────────────────────────┐
│  セクション別並列実行                    │
│                                     │
│  Claude  → Why me / VRIO / CF / 戦略  │
│  Gemini  → Why now / 市場成長性（web検索）│
│  GPT     → オペレーション蓋然性 / 異論    │
└─────────────────────────────────────┘
        ↓
[Claude judge: 統合・合成・最終評価出力]
```

### Judge設計（重要）
Claude-as-judgeには**自己回答優遇バイアス**があるため、以下をシステムプロンプトに明示：

```
- 自分の初回回答を優遇しないこと
- GeminiとGPTの反論・補足を真剣に採用すること
- 異なる視点が正しい場合は明示的に採用し、理由を述べること
```

### モデル分担詳細

| セクション | 主担当 | 理由 |
|---|---|---|
| Why me評価 | Claude | 構造化論理・フレームワーク適用 |
| Why now / 市場成長性 | Gemini | リアルタイム市場データ・web検索 |
| VRIO / マーケ評価 | Claude | フレームワーク精度 |
| オペレーション蓋然性 | GPT | 実務ケース引き出し量 |
| NPV / CF / EXIT | Claude + GPT壁打ち | 数値整合性のクロスチェック |
| クロスSWOT / 打開策 | Claude judge | 統合判断 |

### API設計
- **OpenRouter不使用**（5.5%フィー回避）
- 各プロバイダー直接接続（Anthropic / Google / OpenAI）
- セクション分割 → 並列呼び出し → judge統合
- クローズドネットワーク前提（認証必須）
